# Wazuh Agent
---
## Basic File Integrity Monitoring
On the Windows Server machine, navigate to `C:\Program Files (x86)\ossec-agent`  

![](SS/SS-03122026_1.png)  
Create a backup of the `ossec.conf` file. This ensures that if we break something in the configuration, we can revert back to the default configuration.  

Open the `ossec.conf` file using Notepad.exe.  
 ![](SS/SS-03122026_2.png)
![](SS/SS-03122026_4.png)  
Change the `<frequency>` to 30 seconds as this will send updates to the Wazuh server in 30 second intervals.  
Add a new directory for the File Integrity Monitoring to monitor.  
```
<directories>C:\Users\Public</directories>
```
We will use the `Public` user profile for this example.

Save the file.  

![](SS/SS-03122026_5.png)  
Anytime configuration changes are made, we must restart the Wazuh service.  

![](SS/SS-03122026_6.png)  
Let's simulate a malicious actor creating a new file in the Public Pictures folder.  

![](SS/SS-03122026_7.png)  
Head over to the Wazuh File Integrity Monitoring Events, we can observe that it successfully logged that a new file `evil-malware.txt` was created in the system.

![](SS/SS-03122026_8.png)  
![](SS/SS-03122026_9.png)  
![](SS/SS-03122026_10.png)  
Let's do another simulation where the malicious actor modifies our file `bank-accounts.txt`. The previous value was `12345` and the malicious actor changed it to a new value `b4d-act0r`.  
We can observe in Wazuh that it was also able to log that the file was modified and its integrity checksum changed.  

## Real-time FIM
![](SS/SS-03122026_11.png)  
Let's change the `ossec.conf` frequency back to its original value but this time, in the our directories we will add the parameter `realtime="yes"`.  
Save and restart the Wazuh service.  

![](SS/SS-03122026_12.png)  
Let's create a new file in the Public directory called `b4d-mA1wAre.txt`.  

![](SS/SS-03122026_13.png)  
In Wazuh, we can see more events occurred. First, a new text document was added to the system then it was deleted and re-added because we renamed the file to `b4d-mA1wAre`.  

## Whodata FIM
The whodata option provides more logging capabilities by logging what process and what user changed the file.  

![](SS/SS-03122026_14.png)  
Navigate to the `ossec.conf` file and change the option from `realtime` to `whodata`.  
Save and restart the Wazuh service.  

![](SS/SS-03122026_15.png)  
In the `b4d-mA1wAre.txt` file, lets change the text to "I am bad actor haha."

![](SS/SS-03122026_16.png)  
![](SS/SS-03122026_17.png)  
Navigating back to the Wazuh File Integrity Monitoring, we can expand on the new event and take note that this time, Wazuh logged the `proccess id`, `proccess name`, `user id`, and `user name`.  

![](SS/SS-03122026_18.png)  
Previously without using `whodata`, we can observe that Wazuh did not log those information.

## Report Change FIM
The `report_changes` option will provide further logging capabilities by telling us what data was changed.

![](SS/SS-03122026_19.png)  
Navigate to the `ossec.conf` file and add the option `report_changes="yes"`.  
Save and restart the Wazuh service.  

![](SS/SS-03122026_20.png)  
Let's create a new file called `top-secret.txt` and in the file contains the text `Password: easypass123`.

![](SS/SS-03122026_21.png)  
Say a malicious actor compromised the system and changed the data in the `top-secret.txt` file to now include `H4CKED I GOT YOUR PASSWORD`.

![](SS/SS-03122026_22.png)  
![](SS/SS-03122026_23.png)  
By checking the first integrity change event, we can observe our original data that we added and Wazuh logged it.

![](SS/SS-03122026_24.png)  
When the malicious actor changed our `top-secret.txt` file, Wazuh logged what changed.
